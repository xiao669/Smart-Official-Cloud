# 多场景数据表隔离方案

## 概述
为了实现不同使用场景的数据完全隔离，系统采用多表策略，每个场景使用独立的数据表。

## 场景与表名映射

### 1. 药品管理 (medicine)
- 物品表: `medicine_items`
- 批次表: `medicine_batches`
- 分类表: `medicine_categories`

### 2. 库存管理 (inventory)
- 物品表: `inventory_items`
- 批次表: `inventory_batches`
- 分类表: `inventory_categories`

### 3. 食品管理 (food)
- 物品表: `food_items`
- 批次表: `food_batches`
- 分类表: `food_categories`

### 4. 设备管理 (equipment)
- 物品表: `equipment_items`
- 批次表: `equipment_batches`
- 分类表: `equipment_categories`

### 5. 文档管理 (document)
- 物品表: `document_items`
- 批次表: `document_batches`
- 分类表: `document_categories`

### 6. 自定义场景 (custom)
- 物品表: `custom_items`
- 批次表: `custom_batches`
- 分类表: `custom_categories`

## 数据库设计

### 用户表新增字段
```sql
ALTER TABLE users ADD COLUMN current_mode VARCHAR(20) DEFAULT 'medicine' NOT NULL COMMENT '当前使用场景';
```

### 表结构说明
所有场景的表结构完全相同，只是表名不同：

#### 物品表 (items)
- id: 主键
- name: 物品名称
- code: 物品编码（唯一）
- barcode: 条形码
- category_id: 分类ID
- specification: 规格
- unit: 单位
- manufacturer: 生产厂家
- price: 单价
- description: 描述
- is_deleted: 软删除标记
- created_at: 创建时间
- updated_at: 更新时间

#### 批次表 (batches)
- id: 主键
- item_id: 物品ID（外键）
- batch_number: 批次号
- quantity: 数量
- production_date: 生产日期
- expiry_date: 有效期
- inbound_date: 入库时间
- zero_stock_date: 零库存时间
- remark: 备注
- created_at: 创建时间

#### 分类表 (categories)
- id: 主键
- name: 分类名称
- description: 描述
- sort_order: 排序
- created_at: 创建时间

## 实现方案

### 1. 后端实现
- 创建动态表名工具函数 `get_table_name(mode, base_table)`
- 修改模型层，支持动态表名
- 修改服务层，根据用户的 `current_mode` 选择对应表
- API层传递用户的场景信息

### 2. 前端实现
- App登录后获取用户的 `current_mode`
- 切换场景时调用API更新用户的 `current_mode`
- 所有API请求自动携带当前场景信息

### 3. 数据迁移
- 创建Alembic迁移脚本
- 为每个场景创建对应的数据表
- 将现有 `medicines`、`batches`、`medicine_categories` 数据迁移到 `medicine_items`、`medicine_batches`、`medicine_categories`

## 优势
1. **数据完全隔离**：不同场景的数据互不影响
2. **灵活扩展**：新增场景只需创建新表
3. **性能优化**：每个场景的数据量更小，查询更快
4. **安全性高**：场景间数据无法混淆

## 注意事项
1. 切换场景后，用户只能看到当前场景的数据
2. 不同场景的数据不能共享
3. 需要为每个场景单独配置初始分类数据
4. 数据导入导出需要区分场景
